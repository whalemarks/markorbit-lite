# Application Charter

## Vision

Make MarkOrbit Lite the daily operating workspace for trademark agents using Mo capabilities.

## Mission

Help agents decide what to publish, what materials to use, which trademarks to recommend, which clients to quote, and which leads to follow up.

## Target Users

- Chinese trademark agents;
- intellectual property consultants;
- trademark agency sales and process staff;
- agency administrators;
- platform operators supporting Lite content, templates, services, and prices.

## Product Principles

- Workspace first.
- Daily action over feature menus.
- Private user data by default.
- Mo-aligned, but not blocked by unfinished Mo platform layers.
- Business outcomes over token consumption.
- Simple implementation before generalized infrastructure.

## What Lite IS

- A clean-start Mo application layer.
- A trademark-agent workspace.
- A content, asset, trademark, quote, and lead execution tool.
- A product that reserves future Brain, Runtime, and Workflow compatibility.

## What Lite IS NOT

- Not MarkOrbit V1 continued under a new name.
- Not the full Mo Architecture.
- Not a complete Mo Brain client.
- Not a global trademark data platform.
- Not a workflow engine.
- Not an official filing, payment, escrow, or marketplace system.

## Relationship with Mo

Lite is an application that consumes and exposes Mo capabilities as they become available. It should keep stable boundaries to Mo Data, Brain, Intelligence, Capability, Skill, Workflow, Guide, Flow, and future Runtime.

## Relationship with V1

V1 is a source of validated business lessons, not a source of implementation inheritance. Lite may reuse ideas selectively, but it must not copy V1 structure by default.

## Development Philosophy

Start with governance, then build the smallest coherent workspace loops. Prefer explicit boundaries, private defaults, replaceable services, and documented decisions.

## Release Strategy

Release Lite through foundation-first increments:

1. establish documents and architectural governance;
2. define application boundaries and module contracts;
3. implement workspace-first user loops;
4. reserve Brain and Workflow integration points;
5. connect future Mo platform layers only when stable and authorized.
